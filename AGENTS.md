## 核心开发守则：Production-Ready Guidelines

以下 5 项为当前项目的最高优先级工程约束，后续生成的代码必须无条件遵守。

### 1. 绝对的跨平台与配置隔离
- 禁止硬编码路径。
- Python 路径处理必须统一使用 `pathlib.Path`，例如 `BASE_DIR / "data" / "jobs"`。
- 严禁使用字符串拼接路径，例如 `"data/" + filename` 或 `os.path.join` 与硬编码混用造成平台差异。
- 所有可变配置必须通过 `.env` 注入，并由 `pydantic-settings` 或 `os.getenv` 读取。
- 严禁在代码中写死端口、IP、密码、密钥、数据库连接串、Redis 地址、对象存储地址。

### 2. 数据库无关性
- MVP 阶段允许使用 SQLite。
- 所有数据库访问必须通过 `SQLAlchemy` 或 `SQLModel` ORM 完成。
- 严禁编写依赖某一数据库方言的原生 SQL。
- 数据模型、查询逻辑、迁移策略都必须以未来平滑迁移到 PostgreSQL/MySQL 为前提。

### 3. 无状态架构与存储抽象
- Worker 必须设计为无状态。
- 任务执行产生的业务结果必须写回数据库、消息系统或统一存储服务，禁止永久保存在 Worker 本地磁盘。
- 涉及用户上传代码、结果图表、导出文件等持久化文件时，必须通过统一的 `StorageService` 抽象访问。
- `StorageService` 的接口设计必须允许后续仅替换实现类，即可从本地文件系统切换到 Aliyun OSS、AWS S3 等对象存储。

### 4. 容器化优先
- 所有组件在设计上都必须假设未来运行于彼此独立的 Docker 容器中。
- 前端、FastAPI、消息代理、Worker、量子执行引擎之间只能通过网络或消息系统通信。
- 严禁依赖共享本地磁盘、共享内存或进程内单例跨服务传递状态。
- 开发期允许采用本地进程运行应用代码，但架构设计不得绑定这种运行方式。

### 5. 企业级代码质量
所有新增或修改的 Python 代码必须满足以下要求：
- 严格遵守 PEP 8。
- 强制包含完整的 Type Hints。
- 主要类、函数、API 路由必须包含规范的 Docstring，推荐 Google 或 Sphinx 风格。
- 异常捕获必须具体化，禁止裸 `except:`。
- 必须使用标准 `logging` 模块输出结构化、可检索的日志。
- 优先保持函数短小、职责单一、依赖清晰。

## 架构与实现约束

### 后端约束
- API 层只负责协议适配、鉴权、参数校验和响应组装，不承载核心业务逻辑。
- 业务逻辑放入 `services/`、应用服务层或更明确的领域层中。
- 数据访问统一通过 ORM 会话与模型完成。
- 配置集中管理，避免在模块内部直接散落读取环境变量。
- 后台任务体系按 Celery Worker 的无状态模型设计，不继续强化 RQ 绑定。

### 前端约束
- 页面层负责路由和组合，不承载复杂业务流程。
- feature 内部封装自身的应用逻辑、状态管理和业务组件。
- 通用组件不得依赖具体 feature 的内部实现。
- 接口契约、状态管理和展示逻辑要显式分层，避免混杂。

### 存储与文件约束
- 任何业务文件持久化都必须通过存储抽象层。
- 不得让多个服务隐式依赖同一路径下的文件来协作。
- 临时文件可以存在于本地，但必须可回收、可替换、不可作为系统长期真相源。

以下旧约束视为失效，不得再作为实现依据：
- “数据库固定为 SQLite 文件路径 `backend/app.db`”
- “任务队列固定为 Redis + RQ”

- “可以省略 Type Hints、Docstring、结构化日志”

## 代码质量原则
- 严格遵循 SOLID、KISS、DRY、YAGNI。
- 优先单一职责，避免上帝对象、超大组件、超长函数。
- 新增抽象必须有现实收益，避免为未来不确定需求过度设计。
- 重构优先消除重复、收紧边界、降低耦合。

## 验证要求
完成修改后，至少验证受影响范围：
- 后端改动：运行相关 `pytest` 用例。
- 前端改动：运行相关 `vitest` 用例和 `npm run build`。
- 无法执行验证时，必须明确说明原因、风险和替代验证方式。
