## 核心开发守则：Production-Ready Guidelines

以下 5 项为当前项目的最高优先级工程约束，后续生成的代码必须无条件遵守。

### 1. 绝对的跨平台与配置隔离
- 禁止硬编码路径。
- Python 路径处理必须统一使用 `pathlib.Path`，例如 `BASE_DIR / "data" / "jobs"`。
- 严禁使用字符串拼接路径，例如 `"data/" + filename` 或 `os.path.join` 与硬编码混用造成平台差异。
- 所有可变配置必须通过 `.env` 注入，并由 `pydantic-settings` 或 `os.getenv` 读取。
- 严禁在代码中写死端口、IP、密码、密钥、数据库连接串、Redis 地址、对象存储地址。

### 2. 数据库策略（PostgreSQL 优先，保留迁移能力）
- 当前项目主数据库为 PostgreSQL；开发与联调默认以 PostgreSQL 为基线，不再以 SQLite 作为主运行路径。
- 所有数据库访问必须通过 `SQLAlchemy` / `SQLModel` ORM 完成。
- Schema 变更必须通过 Alembic 管理；禁止把 `create_all()` 作为生产或联调环境的 schema 演进手段。
- `DATABASE_URL` 是唯一运行时数据库入口；禁止在业务代码中分叉出隐式数据库连接来源。
- SQLite 仅允许用于测试、临时本地验证、历史数据迁移脚本输入（如 `SQLITE_SOURCE_DATABASE_URL`）；不得作为新功能验收的默认目标库。
- 禁止在业务层引入新的 SQLite 专属分支或方言锁定写法；若因性能或迁移需要使用方言特性，必须封装在基础设施层并补充回归测试。

### 3. 无状态架构与存储抽象
- Worker 必须设计为无状态。
- 任务执行产生的业务结果必须写回数据库、消息系统或统一存储服务，禁止永久保存在 Worker 本地磁盘。
- 涉及用户上传代码、结果图表、导出文件等持久化文件时，必须通过统一的 `StorageService` 抽象访问。
- `StorageService` 的接口设计必须允许后续仅替换实现类，即可从本地文件系统切换到 Aliyun OSS、AWS S3 等对象存储。

### 4. 容器化优先
- 所有组件在设计上都必须假设未来运行于彼此独立的 Docker 容器中。
- 前端、FastAPI、消息代理、Worker、量子执行引擎之间只能通过网络或消息系统通信。
- 严禁依赖共享本地磁盘、共享内存或进程内单例跨服务传递状态。
- 开发期允许采用本地进程运行应用代码，但架构设计不得绑定这种运行方式。

### 5. 企业级代码质量
所有新增或修改的 Python 代码必须满足以下要求：
- 严格遵守 PEP 8。
- 强制包含完整的 Type Hints。
- 主要类、函数、API 路由必须包含规范的 Docstring，推荐 Google 或 Sphinx 风格。
- 异常捕获必须具体化，禁止裸 `except:`。
- 必须使用标准 `logging` 模块输出结构化、可检索的日志。
- 优先保持函数短小、职责单一、依赖清晰。

## 架构与实现约束

### 后端约束
- API 层只负责协议适配、鉴权、参数校验和响应组装，不承载核心业务逻辑。
- 业务逻辑放入 `services/`、应用服务层或更明确的领域层中。
- 数据访问统一通过 ORM 会话与模型完成。
- 配置集中管理，避免在模块内部直接散落读取环境变量。
- 后台任务体系按 Celery Worker 的无状态模型设计，不继续强化 RQ 绑定。
- 任何用户态业务查询必须显式收敛到 `tenant_id + user_id` 边界；越权场景统一返回 `404`。
- 任务链路必须保持边界清晰：
  - 代码任务：`worker -> execution-service -> Docker runner`
  - 图形化电路任务：`circuit-worker -> qibo hot executor`
- 禁止将图形化电路任务回退为“拼接 Python 脚本并复用代码任务链路”的实现。
- `execution-service` 是内部服务边界，不对前端暴露；其自身禁止配置为 `remote` 执行后端。
- 修改任务执行逻辑、门映射、payload 校验或 Worker 依赖后，必须重启对应 Celery Worker；不得假设自动热重载生效。
- 任务提交必须保持幂等语义：`Idempotency-Key` 在租户和用户维度内唯一绑定任务，禁止破坏去重契约。

### 前端约束
- 页面层负责路由和组合，不承载复杂业务流程。
- feature 内部封装自身的应用逻辑、状态管理和业务组件。
- 通用组件不得依赖具体 feature 的内部实现。
- 接口契约、状态管理和展示逻辑要显式分层，避免混杂。
- 图形化工作台必须明确区分“本地预览结果”和“后端正式任务结果”，禁止混淆展示语义。

### 存储与文件约束
- 任何业务文件持久化都必须通过存储抽象层。
- 不得让多个服务隐式依赖同一路径下的文件来协作。
- 临时文件可以存在于本地，但必须可回收、可替换、不可作为系统长期真相源。

- Python 3.11
- Node.js 18+
- PostgreSQL 16（默认主数据库）
- Redis（消息代理/状态协作基础设施）

## 明确替换的旧约束
以下旧约束视为失效，不得再作为实现依据：
- “SQLite 是 MVP 默认主数据库”
- “可以通过运行时 `create_all()` 管理线上或联调 schema”
- “任务队列固定为 Redis + RQ”
- “图形化任务可与代码任务共享同一执行链路”
- “execution-service 可使用 remote 作为自身执行后端”
- “Worker 依赖本地磁盘保存业务状态”
- “开发方式可默认假设所有服务共享宿主机文件系统”
- “可以省略 Type Hints、Docstring、结构化日志”

## 代码质量原则
- 严格遵循 SOLID、KISS、DRY、YAGNI。
- 优先单一职责，避免上帝对象、超大组件、超长函数。
- 新增抽象必须有现实收益，避免为未来不确定需求过度设计。
- 重构优先消除重复、收紧边界、降低耦合。

## 验证要求
完成修改后，至少验证受影响范围：
- 后端改动：运行相关 `pytest` 用例。
- 前端改动：运行相关 `vitest` 用例和 `npm run build`。
- 涉及任务链路改动：至少补充一次 API/worker 端到端冒烟验证。
- 无法执行验证时，必须明确说明原因、风险和替代验证方式。

## 文档要求
- 新增架构约束、运行方式和依赖关系时，必须同步更新相关文档。
- 文档内容必须与当前代码和脚本行为一致。
- 若代码现状与本文件冲突，应优先修正代码或显式记录技术债，不得继续复制旧约束。
