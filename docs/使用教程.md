# QCP MVP 使用教程（Windows + uv + Celery）

本教程用于在本地完整跑通 `mvp_qcp`：前端 Web IDE、后端 FastAPI、Redis 队列、Celery Worker、Qibo 执行。

## 1. 前置要求

- 操作系统：Windows（PowerShell）
- Python：3.11（项目固定）
- Node.js：18+
- 包管理：`uv`、`npm`
- Redis：本地可访问 `127.0.0.1:6379`

项目根目录：

```powershell
cd "E:/02_Projects/quantuncloudplatform/mvp_qcp"
```

## 2. 安装依赖

### 2.1 后端依赖（uv + Python 3.11）

```powershell
cd "E:/02_Projects/quantuncloudplatform/mvp_qcp/backend"
uv venv --python 3.11
uv pip install -r requirements.txt
```

### 2.2 前端依赖

```powershell
cd "E:/02_Projects/quantuncloudplatform/mvp_qcp/frontend"
npm install
```

## 3. 启动 Redis

确保 Redis 服务已启动并监听 `127.0.0.1:6379`。

常见方式（任选其一）：

- 已安装 Windows 服务：启动对应 Redis 服务。
- 便携版：在 Redis 目录执行 `redis-server.exe`。

快速检查端口：

```powershell
Test-NetConnection 127.0.0.1 -Port 6379
```

`TcpTestSucceeded` 为 `True` 表示 Redis 可用。

## 4. 启动项目

### 方式 A：一键启动（推荐）

在项目根目录执行：

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File "scripts/start-dev.ps1"
```

该脚本会：

1. 检查 `uv` / `npm`
2. 检查 Redis 连通性（若未启动则自动尝试启动 Redis 服务或 `redis-server`）
3. 检查后端 Python 是否为 3.11
4. 检查后端依赖（fastapi/sqlmodel/redis/rq/qibo）
5. 检查前端 `node_modules`
6. 启动 API（8000）
7. 启动 Celery Worker
8. 启动前端（5173）

### 方式 B：手动三终端启动

终端 1（后端 API）：

```powershell
cd "E:/02_Projects/quantuncloudplatform/mvp_qcp/backend"
uv run uvicorn app.main:app --reload --host 0.0.0.0 --port 8000
```

终端 2（Celery Worker）：

```powershell
cd "E:/02_Projects/quantuncloudplatform/mvp_qcp/backend"
uv run celery -A app.worker.celery_app:celery_app worker --loglevel=info --pool=solo
```

终端 3（前端）：

```powershell
cd "E:/02_Projects/quantuncloudplatform/mvp_qcp/frontend"
npm run dev
```

## 5. 健康检查

日常稳定性基线（推荐每天先跑一次）：

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File "scripts/daily-baseline.ps1"
```

仅检查版本与运行时，不做联调检查：

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File "scripts/daily-baseline.ps1" -SkipHealth
```

### 5.1 基础健康检查

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File "scripts/dev-health-check.ps1"
```

检查项：

- Redis 端口
- API `GET /api/health`
- Frontend 页面可访问

### 5.2 深度链路检查（含注册/登录/提交任务）

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File "scripts/dev-health-check.ps1" -Deep
```

## 6. 前端使用流程

1. 打开 `http://127.0.0.1:5173`
2. 进入注册页，创建账号（当前字段为 `username + password`）
3. 登录后进入任务页
4. 默认代码已提供 Bell 态示例（Qibo）
5. 点击“提交任务”
6. 查看任务状态（可自动轮询）
7. 成功后获取结果并查看概率柱状图（`00/11` 接近 50/50）

## 7. 当前 API 入口

- 健康检查：`GET /api/health`
- 注册：`POST /api/auth/register`
- 登录：`POST /api/auth/login`
- 提交任务：`POST /api/tasks/submit`
- 查询状态：`GET /api/tasks/{task_id}`
- 获取结果：`GET /api/tasks/{task_id}/result`

后端默认地址：`http://127.0.0.1:8000`。

## 8. 常见问题排查

### 8.1 前端注册/登录报 `Failed to fetch`

原因通常是跨域或后端未启动。

确认：

- 后端已启动并可访问 `http://127.0.0.1:8000/api/health`
- 后端已启用 CORS（项目已配置 `5173`）

### 8.2 Worker 报 `os.fork` 或 `SIGALRM` 相关错误

这是 Windows 上 RQ 默认 Worker 的兼容问题。项目已改为 `WindowsSimpleWorker + TimerDeathPenalty`，应使用：

```powershell
uv run celery -A app.worker.celery_app:celery_app worker --loglevel=info --pool=solo
```

不要用旧的 Celery 启动方式。

### 8.3 `Failed to spawn: celery`

项目已从 Celery 切换到 RQ，此报错说明你仍在执行旧命令。请改用第 4 节命令。

### 8.4 `ModuleNotFoundError: sqlmodel`

后端依赖未安装或未在 `backend` 目录执行 `uv` 命令。重新执行第 2.1 节安装步骤。

### 8.5 不是 Python 3.11

脚本会拒绝启动。请重建环境：

```powershell
cd "E:/02_Projects/quantuncloudplatform/mvp_qcp/backend"
uv venv --python 3.11
uv pip install -r requirements.txt
```
