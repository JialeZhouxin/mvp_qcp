# 中国网络环境下 Docker 拉起与依赖供应链风险记录（2026-04-02）

## 背景

项目目标包含“未来在中国环境上线部署”。在开发阶段已出现以下问题：

- `docker compose up --build` 受网络影响，镜像拉取与依赖下载不稳定。
- 部分镜像/依赖在国内网络路径下可达性与速度不确定。
- 尚未建立内网私有制品仓库（镜像、Python、Node 依赖均未完全内网化）。

本文档用于记录风险与后续行动，不作为当前系统事实来源。

## 问题现象

### 1. Docker 相关

- 基础镜像和服务镜像默认来自公网源（如 Docker Hub）。
- 开发机网络波动时，镜像拉取和构建容易超时。

### 2. Python / Node 依赖相关

- Docker 构建阶段需要在线安装 Python 与 npm 依赖，受公网源可用性影响。
- 依赖“能否在国内稳定获取”缺少持续审计机制。

### 3. 上线导向风险

- 若运行环境要求严格出网控制，当前路径无法保证“零公网依赖”。
- 仅依赖临时加速器，不具备长期稳定性与可审计性。

## 当前状态（已落地的缓解）

- Compose 与 Dockerfile 已支持通过环境变量注入构建镜像源和依赖源参数。
- 已提供中国/受限网络示例环境文件：`.env.cn.example`。
- 已新增依赖来源审计脚本：`scripts/cn-supply-chain-audit.ps1`。
- 已新增运行 Playbook：`docs/china-deployment-playbook.md`。

说明：以上改造是“可用性与可迁移性”准备，不等于生产上线就绪。

## 风险分级

| 风险项 | 等级 | 影响 |
|---|---|---|
| 公网镜像源不稳定导致构建失败 | 高 | 开发/联调阻塞，CI 失败率上升 |
| 依赖来源不可控（临时切换源） | 高 | 构建不可复现，供应链风险增加 |
| 缺少内网仓库导致无法零出网 | 中 | 不满足后续生产网络策略 |
| 源切换无统一门禁与审计 | 中 | 团队配置漂移，排障成本上升 |

## 后续行动建议（待排期）

1. 短期：统一团队 `.env` 与构建参数基线，先保证开发环境可稳定拉起。
2. 中期：建设最小私有镜像仓库，先托管项目必需镜像，再逐步扩展。
3. 中期：将供应链审计脚本接入 CI，禁止未批准的 registry/host 进入主干。
4. 长期：落地镜像与依赖全链路内网化，支持受限网络与合规要求。
